还是我R2的时候写的。。自己寻找子问题然后逐个予以突破的感觉太好了。。。。

接上回。。。

上回说到我否决了两个方向。。
( 分析蛋糕结构。。和生成多种绕线方案。。

那么算法主框架。。。

初始生成绕线方案。。
（任何一种绕线方案都可以一次性解决 disjoint 的问题。。（首先要解决的问题。。）。。并且可以使用二分查找。。。
。。。然后这里再采用螺旋绕线方案。。（因为可以最大限度的保留 rim 的加分。。。同时比较有效的保留 rose 的加分。。。

之后开始二分答案。。。

下面开始研究切蛋糕函数。。
f(x) 。。表示是否可以切成最小面值为 x 的蛋糕。。。

方案1：
REP(i, G){
REP(j, C) 找到消耗蛋糕块数最少的切割。。。
}

方案2：
。。对每个人，生成一个偏爱蛋糕的序列。。做成 Rank 。。
REP(i, G){
REP(j, C)  如何 第 i 个人可以用 第 j 块蛋糕填，break;
}

方案3：
回溯搜索。。
REP(i, G){
REP(j, C) 回溯。。
}

方案4：
REP(i, G){
枚举所有 (人, 蛋糕) 二元组，从中选择消耗最少的。。
}

——————
以上除了第四种方案其他都是依赖于 人的排列 的。。（而这个解空间巨大 （  40! 。。。
。。。同时对两个排列之间。。暂无什么有效的方法立刻分析出好坏。。
。。然后经测试第四种方案的效果也是最差的。。。


下面着重分析前三种方案。。每种方案都配套一定的改进措施。。

首先就方案2，主要依赖于 Rank 的选择。。。

这里有好几种备选方案。。

1. 依据整块蛋糕的权值。。
2. 依据绕线前 S*4 个。。。（更有效的利用 Rim 的启发信息。。
3. 对于前面采用 S*4 作为 Rank 后期采用 SQR(S) - S * 4 ( 前期优先利用 Rim 。。。后期优先考虑 Base 。。。
4. 根据过往频率予以扰动。。（因为我认为对于每个人。。在优秀方案中。。其可能分配的方案只是蛋糕的一个子集。。。

实际操作中使用 2 和使用 1 的效果差别太大了。。

3 的作用不是很明显。。
4 起到了反作用。。（感觉是我利用的方式不对。。。


然后对方案1 。。（。看上去方案1 要多一个 C 的因子常数。。但是实际中好像并不是特别明显。。。。

对方案1 有一个很有效的改进。。如果一轮贪心失败。。则将失败的位置移动到序列的开始位置。。再 Run 一遍切割。。
。。。该位移可在切割过程中反复进行。。。设置一个最大失败次数的上界。。。

使用这个优化的算法在 Arean 中跑出了 9800 的得分。。。并且实现调整到 100s 后仍有很大改进空间。。。
我以这个版本 100s 的程序作为最优解写了本地的 Test 。。。。


对于方案 2。。。
有一个我一开始以为会很有效的改进。。
是用栈记录每个蛋糕上一步的匹配位置。。如某一步发生匹配失败，则可以进行替换操作。。
（替换过程可能仍然要替换。。。于是出现递归。。。。

实际可能需要对这个深度予以控制。。。
但是实际编码后发现效果并不理想。。。


总之方案1 和 方案2 大概是同一个等级的。。。


方案3 。。。

方案1 和 方案2 都可以看做是 禁止回溯版本的 方案 3.。。
那两个优化也不过是加入了搜索的成分。。。


在讨论方案3 之前。。。不得不说一下方案1 和方案2 的一个巨大问题。。。。
。。就是用它们做 f() 函数不是单调的。。。

。。。具体分析的话应该是属于那种 整体单调。。局部不单调 的函数。。。（所以二分答案才会仍然有效。。。

至于局部不单调的原因。。。方案1 和 方案2 都有问题。。
比如在 X+1 的可能能得到合法解。。但是在 x 更小的时候。。可能因为一些年轻时犯下的错误导致后面贪心构造失败。。。。
因此在设计新的 f() 函数时。。必须把这种稳定性也纳入考虑范围。。。。


假设我们可以回溯所有方案的话。。那么 f() 函数必然单调。。。。
但是回溯所有方案压力太大。。这里使用先贪心再搜索的方法。。只在最后几层允许回溯。。


。。。至此为止决定舍弃方案1 和方案2.。。。集中精力修改完善方案3.。。

理由如下：

无法理性的判断方案1 和方案2 的好坏。。（所以干脆都不要。。。
方案3 的回溯过程更可控。。。后期可以优化的地方也多。。。
总的思路就是先贪心。。随着深度的提高并且可行域变得有限之后开始搜索。。。
